1한눈에 보는 개요

AgentPod는 호스트에서 도는 Python CLI(오케스트레이터)와 프로젝트마다 하나씩 뜨는 격리된 Docker 컨테이너로 이루어집니다. 위험한 자율 실행 플래그(--dangerously-skip-permissions)는 컨테이너 안에서만 돌고, 컨테이너 밖 호스트에서는 절대 실행되지 않습니다. 필요한 모든 것(인증·컨텍스트)은 호스트 상태를 긁어오지 않고 파일과 바인드 마운트로 선언합니다.

1 : 1프로젝트 : 컨테이너 (경로 해시로 결정)
refcount마지막 세션이 나가면 자동 stop
docker exec컨테이너는 대기, 도구는 붙어서 실행
0 sockdocker.sock 마운트 없음 = 격리 유지

2움직이지 않는 설계 원칙

격리가 자율성을 가능케 한다

무인 실행 플래그의 폭발 반경을 컨테이너로 가둔다. 격리 없이 이 플래그를 쓰는 것은 재앙.

호스트 독립 (선언적)

env·컨텍스트를 호스트에서 자동으로 긁어오지 않는다. 파일에 적어두면 갖춰진다 → 어디서 실행해도 동일.

컨테이너는 일회용, 상태는 밖에

컨테이너는 언제든 버릴 수 있고, 로그인·캐시 등 무거운 상태는 바인드 마운트로 영속.

docker는 바깥에서만

컨테이너를 스폰·관리하는 docker 명령은 신뢰할 수 있는 호스트 오케스트레이터만 실행. 에이전트 컨테이너 안엔 docker가 없다.

3모듈 구조

오케스트레이터는 한 파일 = 한 책임으로 잘게 나뉜 Python 패키지입니다. 순수 함수 모듈(naming/paths/registry/context)과 부수효과 모듈(session/docker_ctl)을 CLI가 조립합니다.

src/agentpod/ ├── cli.py # Typer 앱: build/run/shell/status/stop/rm/context/export 조립 ├── naming.py # 경로 → 결정적 projectId / 컨테이너명 / 락 prefix ├── paths.py # ~/.agent 데이터 레이아웃 + ensure_layout() ├── registry.py # 도구 정의(claude): 바이너리·플래그·크레덴셜 마운트 ├── context.py # 프로젝트별 MD 컨텍스트 폴더 → 마운트 해석 ├── plugins.py # 호스트 사전 설치(seed_superpowers) + export_skills/export_mcp_servers ├── session.py # 락파일·레퍼런스 카운팅·크래시 복구·시그널 └── docker_ctl.py # docker CLI subprocess 래퍼 + Mount 데이터클래스 Dockerfile · docker/agent-entrypoint.sh # 에이전트 컨테이너 이미지 + 초기화
모듈대표 함수역할
namingproject_id, container_name, lock_prefix경로를 안정적인 식별자로
pathsagent_root, ensure_layout~/.agent 하위 디렉토리 보장
registryget_tool, REGISTRY도구별 차이를 데이터로 (개방-폐쇄)
contextresolve_mount컨텍스트 폴더 존재 시 ro 마운트 반환
pluginsseed_superpowers, export_skills, export_mcp_servers호스트 사전 설치 + 설치 상태를 프로젝트 파일로 공유
sessioncreate_lock, active_sessions, release_lock세션 카운팅으로 컨테이너 stop 제어
docker_ctlrun_detached, exec_interactive, container_statedocker CLI를 얇게 감쌈

4실행 흐름 — agentpod run

사용자가 프로젝트 디렉토리에서 agentpod run을 치면 벌어지는 일:

1
이름 계산 — 현재 경로를 realpath로 정규화 → projectId = basename-sha256[:12] → 컨테이너명 agent-<projectId>.
이미지 보장 — agentpod:latest가 없으면 자동 빌드.
2
컨테이너 스폰/재사용 — 상태가 running이면 재사용, exited면 start, 없으면 docker run -d로 새로 생성(마운트·workdir·env-file 조립).
초기화(entrypoint) — git safe.directory 설정 → MD 컨텍스트가 있으면 ~/.claude/CLAUDE.md에 @import 주입 → superpowers 플러그인 설치·활성화(§7) → 프로젝트 .mcp.json 자동 승인 → agent.toml/skills.toml에 선언된 스킬 설치 → sleep infinity로 대기.
3
세션 락 생성 — ~/.agent/locks/<prefix>--<sessionId>.lock에 현재 PID 기록. 시그널 핸들러 등록.
도구 실행 — docker exec -it로 claude --dangerously-skip-permissions를 컨테이너 안에서 실행(TTY 상속).
4
종료·정리 — 세션 락 삭제 → 같은 prefix에 살아있는 다른 세션이 없으면 컨테이너 stop.
# 런처가 컨테이너 안에서 조립·실행하는 명령 (개념)
docker exec -it agent-<projectId> \
  claude --dangerously-skip-permissions   # ← 이 플래그는 컨테이너 안에서만

5결정적 이름 규칙

같은 프로젝트는 항상 같은 컨테이너 이름이 되도록 경로를 해시합니다. 그래서 다시 들어가면 세션·컨텍스트가 이어지고, claude --continue가 동작합니다.

projectId     = <정규화된 basename>-<sha256(realpath)[:12]>
컨테이너명    = "agent-" + projectId
락 prefix     = projectId            # --profile 명시 시: projectId + "--p--<profile>"

# 예
/home/me/work/MyRepo  →  myrepo-12e2c4ef8799  →  agent-myrepo-12e2c4ef8799

basename은 소문자화하고 영숫자 외 문자를 하이픈으로 축약합니다. 경로는 심볼릭 링크에 흔들리지 않도록 realpath로 정규화한 뒤 해시합니다.

6세션 수명주기 — 레퍼런스 카운팅

한 컨테이너에 여러 세션이 붙을 수 있으므로(향후 병렬 모델의 전제), 컨테이너를 언제 멈출지는 락파일 카운팅으로 결정합니다.

생성

<prefix>--<sessionId>.lock 파일에 PID 기록.

집계

같은 prefix 락을 열거하되 살아있는 PID만 센다.

정리

내 락 삭제 → 남은 활성 세션이 0이면 컨테이너 stop.

크래시 복구 (stale 청소)

오케스트레이터가 강제 종료되면 락파일이 남을 수 있습니다. PID 생존 검사로 유령 락을 즉시 청소합니다.

# PID 생존 판정
os.kill(pid, 0)
  → 성공 / PermissionError  ⇒ 살아있음
  → ProcessLookupError      ⇒ 죽음 → 락 즉시 삭제
시그널 처리. SIGINT / SIGTERM / SIGHUP + atexit로 정리 루틴을 걸고, "이미 정리함" 플래그로 중복 실행을 막습니다. SIGHUP이 없는 플랫폼에서도 등록이 깨지지 않도록 시그널을 이름으로 안전하게 조회합니다.

7크레덴셜 격리 & 인증

사람의 ~/.claude는 절대 건드리지 않습니다. 에이전트 전용 저장소 ~/.agent/...를 컨테이너의 /home/agent/.claude(플러그인·설정)와 /home/agent/.claude.json(로그인)에 각각 바인드 마운트합니다 — 이 둘은 서로 다른 분리 규칙을 따릅니다.

방식메커니즘영속·분리
API 키프로젝트 .env의 ANTHROPIC_API_KEY → --env-file로 주입파일에 선언, 프로젝트별
대화형 로그인컨테이너 안 claude login 1회 → ~/.agent/claude.json(또는 --profile 지정 시 ~/.agent/profiles/<p>/claude.json)에 기록바인드 마운트로 컨테이너가 죽어도 유지. 항상 공유 — 프로젝트별로 분리되지 않음
플러그인/스킬 설치 상태~/.claude 디렉토리 → --profile 없으면 ~/.agent/profiles/<projectId>/claude, 명시하면 ~/.agent/profiles/<profile>/claude기본은 프로젝트별 자동 격리 — 한 프로젝트에서 설치한 플러그인이 다른 프로젝트로 새지 않음. 같은 --profile을 여러 프로젝트에 쓰면 의도적으로 공유

둘은 공존 가능하며, API 키가 있으면 그것이 우선합니다.

호스트 사전 설치 — plugins.seed_superpowers

컨테이너가 뜨기 직전, ensure_container()가 호스트에 있는 claude CLI로 그 프로젝트의 격리된 creds 디렉토리에 superpowers를 미리 설치·활성화합니다(CLAUDE_CONFIG_DIR 환경변수로 대상 지정). 그러면 컨테이너 안 entrypoint의 설치 단계는 대부분 "이미 있음"을 보고 건너뛰어, 매 컨테이너 부팅마다 git clone하지 않아도 됩니다.

best-effort, 검증 후 신뢰: 호스트에 claude가 없으면 조용히 스킵(→ 컨테이너 entrypoint가 대신 설치). 설치가 끝나도 plugins/cache/<marketplace>/superpowers/에 실제 파일이 들어왔는지 검증하고, 불일치하면(예: WSL에서 PATH로 잡힌 크로스-OS claude 바이너리가 CLAUDE_CONFIG_DIR를 잘못 해석해 마켓플레이스 등록만 되고 내용은 못 받아온 경우) 만든 것을 전부 롤백해 컨테이너의 entrypoint가 깨끗하게 재시도하도록 합니다.

팀원과 공유 — agentpod export

설치·연결은 호스트 상태(위 표)에만 남아 git clone으로 안 옮겨집니다. agentpod export는 현재 프로젝트에 설치된 플러그인/스킬을 agent.toml의 [[skills]]로, claude mcp add로 붙인 MCP 서버를 .mcp.json으로 스냅샷합니다 — MCP의 env/헤더 값은 ${VAR} 플레이스홀더로 바뀌고 실제 값은 .env(gitignore)로 분리되어, 시크릿이 커밋 파일에 박히는 일이 없습니다. 커밋해두면 다음 사람은 git clone + agentpod run만으로 재현됩니다.

8컨테이너별 MD 컨텍스트 주입

컨테이너(=프로젝트)마다 자기만의 지침·문서 세트를 봅니다. 폴더 하나 = 컨테이너 하나이며, 다른 컨테이너의 MD를 상속하지 않습니다.

~/.agent/contexts/<projectId>/
├── CLAUDE.md      # 에이전트 지침 (있으면 자동 주입)
└── *.md           # 참고 문서
A
폴더가 존재하면 /home/agent/context에 읽기 전용으로 마운트.
entrypoint가 context/CLAUDE.md를 발견하면 유저 메모리 ~/.claude/CLAUDE.md에 한 줄 추가:
B
@/home/agent/context/CLAUDE.md — Claude가 이 import를 따라 지침을 자동 로드.
핵심: 사용자의 저장소를 전혀 건드리지 않습니다. 주입은 컨테이너 내부 유저 메모리 파일에만 일어나고, 중복 추가되지 않도록 idempotent 합니다. → 파일만 넣으면 끝, 코드 수정 없음.

9컨테이너 이미지

캐시 친화적으로 "거의 안 바뀜 → 자주 바뀜" 순서의 레이어로 구성한 멀티스테이지 빌드입니다.

레이어내용
베이스ubuntu:24.04 + curl git ca-certificates unzip + locales/tzdata
런타임Node.js 22 (claude CLI 및 npx 기반 MCP 구동용)
사용자비루트 agent (uid 1000) — 호스트 사용자와 uid를 맞춰 바인드 마운트 소유권 정합
도구@anthropic-ai/claude-code 전역 설치 → claude on PATH
진입점agent-entrypoint.sh · CMD ["sleep","infinity"]
Ubuntu 24.04 주의: 베이스 이미지에 이미 uid 1000인 ubuntu 사용자가 있어, agent를 uid 1000으로 만들려면 기존 사용자를 먼저 제거합니다(userdel -r ubuntu). 이 값이 WSL2 호스트 사용자(대개 uid 1000)와 맞아야 마운트된 프로젝트 파일 쓰기 권한이 정상 동작합니다.

10보안 모델

불변식상태
자율 플래그는 컨테이너 안에서만보장 호스트 어떤 경로에도 --dangerously-skip-permissions 없음
docker.sock 마운트 없음보장 DooD 미채택 → 컨테이너가 호스트 루트를 얻지 못함
--network host 미사용보장 기본 bridge로 네트워크 격리
호스트 gitconfig/ssh 마운트없음 사람 개인키를 자율 에이전트에 노출 안 함
크레덴셜 위치격리 사람 ~/.claude가 아닌 ~/.agent
플러그인/스킬 설치 상태프로젝트별 격리 기본적으로 다른 프로젝트 컨테이너로 새지 않음(--profile로 명시해야 공유)
비밀 값gitignore .env는 커밋 안 됨, 이미지/저장소에 하드코딩 없음

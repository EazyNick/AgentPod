AgentPod · Phase 1

사용자 가이드

프로젝트마다 격리된 Docker 컨테이너를 띄우고 그 안에서 Claude Code를 실행하는 방법 — 설치부터 인증, 컨테이너별 지침 관리, 문제 해결까지.

요구사항 설치 빠른 시작 명령어 인증 MD 컨텍스트 여러 컨테이너 공유하기 문제 해결 안전 수칙

1요구사항

항목내용
호스트 OSWSL2 / Linux (권장 타깃)
컨테이너 런타임Docker (데몬 실행 중이어야 함)
Python3.12 이상
Windows + WSL2 사용자: 오케스트레이터를 Ubuntu(WSL2) 안에서 돌리려면 Docker Desktop → Settings → Resources → WSL Integration에서 사용하는 배포판(예: Ubuntu)을 켜야 docker 명령이 배포판 안에서 잡힙니다. 또한 배포판에 python3-pip/python3-venv가 필요할 수 있습니다(sudo apt install -y python3-pip python3-venv).

2설치

한 번에 설치 — 시스템 패키지 + agentpod 명령(PATH 등록) + 이미지 빌드까지 자동:

# 저장소 루트에서
./install.sh                 # 빌드까지
./install.sh --no-build      # 이미지 빌드는 나중에

설치 후 현재 셸에서 agentpod가 안 잡히면 새 터미널을 열거나 source ~/.bashrc.

# (선택) 스크립트 없이 수동 설치
python3 -m venv ~/.venvs/agentpod && source ~/.venvs/agentpod/bin/activate
pip install -e ".[dev]"
pytest -m "not integration"
어디서 실행하나? — 소스 폴더와 무관합니다. 설치 후 agentpod는 전역 명령이 됩니다(venv를 쓴다면 그 venv가 활성화된 셸에서). AgentPod 소스가 있는 폴더로 갈 필요가 없습니다. agentpod run은 “지금 있는 디렉토리(cwd)”를 프로젝트로 간주해 컨테이너를 만듭니다. 
# venv를 쓴다면 새 터미널마다 한 번
source ~/.venvs/agentpod/bin/activate
# 아무 프로젝트로 이동 → 그 폴더 기준으로 컨테이너
cd ~/anywhere/my-project
agentpod run
즉 컨테이너는 현재 작업 디렉토리의 경로로 결정됩니다. 소스 위치는 상관없습니다.

3빠른 시작

가장 간단히 — Docker만 (Python·설치 불필요) 
# 1) 이미지 만들기 — AgentPod 저장소 폴더에서 딱 1회
docker build -t agentpod:latest .

# 2) 실행 — 작업할 폴더(예: agents/apply-verify)에서
mkdir -p ~/.agent/claude
docker run --rm -it -v "$PWD":/project -w /project \
  -v ~/.agent/claude:/home/agent/.claude \
  agentpod:latest claude --dangerously-skip-permissions
그 폴더의 CLAUDE.md·.mcp.json을 자동으로 읽고, superpowers도 자동 설치됩니다. 매번 길게 치기 싫으면 별칭 한 줄: 
echo 'agentpod(){ mkdir -p ~/.agent/claude; docker run --rm -it -v "$PWD":/project -w /project -v ~/.agent/claude:/home/agent/.claude agentpod:latest claude --dangerously-skip-permissions "$@"; }' >> ~/.bashrc && source ~/.bashrc
# 이후: 작업 폴더에서   agentpod
간단한 대신 결정적 이름·세션 자동정리·다중 컨테이너 관리 같은 agentpod CLI 편의 기능은 빠집니다. 그 기능이 필요하면 아래 정식 설치를 쓰세요. 또 리소스 제한 기본값도 자동 적용되지 않으니 docker run에 --memory 4g --cpus 2 --pids-limit 512를 직접 붙이세요(§4).
  1. 이미지 빌드 (최초 1회, 몇 분 소요) 
    agentpod build
  2. 프로젝트로 이동 
    cd /path/to/your/project
  3. 인증 준비 — .env에 API 키를 넣거나(§5), 아래 실행 후 컨테이너 안에서 claude login.
  4. 에이전트 실행 — 컨테이너를 스폰/재사용하고 Claude Code 대화형 세션 진입 
    agentpod run
  5. 종료 — 세션을 빠져나오면(Ctrl+D 등) 마지막 세션일 때 컨테이너가 자동으로 멈춥니다.

4명령어 8개

agentpod build [--force]
에이전트 이미지 빌드. 이미 있으면 skip(--force로 강제 재빌드).
agentpod run
현재 프로젝트의 컨테이너를 스폰/재사용 → 대화형 실행. --tool(claude/codex/opencode) · --profile · --memory/--cpus/--pids(아래). 최상위 agentpod --help에는 --tool이 안 보이니 agentpod run --help로 확인하세요.
agentpod shell
같은 컨테이너에 bash로 접속(디버깅·최초 claude login용).
agentpod status
모든 agent-* 컨테이너와 상태, 활성 세션 수 표시.
agentpod stop
현재 프로젝트의 컨테이너를 stop.
agentpod rm
현재 프로젝트의 컨테이너를 stop 후 제거.
agentpod context
현재 프로젝트의 MD 컨텍스트 폴더 경로 출력(없으면 안내).
agentpod export
설치된 claude 플러그인/스킬 + MCP 서버를 agent.toml/.mcp.json에 기록해 팀원과 공유(아래 §8).

stop/rm/context는 인자 없이 쓰면 현재 디렉토리 기준입니다.

리소스 제한 (폭주 방지)

자율 실행이 호스트를 마비시키지 않도록 컨테이너에 메모리·CPU·PID 상한을 겁니다.

항목기본값변경 방법
메모리4g--memory / AGENT_MEMORY
CPU2--cpus / AGENT_CPUS
PID 수512--pids / AGENT_PIDS_LIMIT

① agentpod run으로 실행할 때 — 기본값이 자동 적용되고, 옵션/환경변수로 덮어씀:

# 실행마다
agentpod run --memory 8g --cpus 4 --pids 1024
# 호스트 기본값 (빈 값 = 해당 제한 해제)
export AGENT_MEMORY=8g AGENT_CPUS=4 AGENT_PIDS_LIMIT=1024

② docker run만으로 실행할 때 — agentpod 오케스트레이터를 안 거치므로 기본값이 자동 적용되지 않습니다. docker 플래그를 직접 붙이세요:

docker run --rm -it --memory 4g --cpus 2 --pids-limit 512 \
  -v "$PWD":/project -w /project \
  -v ~/.agent/claude:/home/agent/.claude \
  agentpod:latest claude --dangerously-skip-permissions

위 예시의 ~/.agent/claude는 공용(비분리) 계정 경로입니다. agentpod run을 쓰면 프로젝트마다 자동으로 ~/.agent/profiles/<projectId>/claude로 분리되지만(§ 프로파일 참고), 수동 docker run에서 그 격리를 그대로 흉내내려면 이 경로를 직접 프로젝트별로 바꿔주세요.

주의: 제한은 컨테이너 생성 시점에만 적용됩니다. 이미 떠 있는 컨테이너의 한계를 바꾸려면 agentpod rm(또는 docker rm) 후 다시 실행하세요.

멀티툴 — claude / codex / opencode

--tool로 실행할 AI 코딩 CLI를 고릅니다. 도구별 자율 플래그·크레덴셜 저장소가 분리됩니다.

agentpod run --tool claude       # 기본
agentpod run --tool codex
agentpod run --tool opencode
도구크레덴셜(호스트)자율 플래그(컨테이너 내부)
claude~/.agent/claude--dangerously-skip-permissions
codex~/.agent/codex--dangerously-bypass-approvals-and-sandbox
opencode~/.agent/opencode(기본 없음)

프로파일 — 기본은 프로젝트별 자동 분리

--profile을 안 쓰면, 플러그인/스킬 설치 상태가 프로젝트 폴더마다(project_id 기준) 자동으로 격리됩니다. 한 프로젝트 컨테이너에서 설치한 플러그인이 다른 프로젝트 컨테이너로 새지 않습니다. 단, 로그인(계정 인증)은 프로파일과 무관하게 항상 공유됩니다 — 한 번 로그인하면 모든 프로젝트에서 그대로 씁니다.

agentpod run                     # 기본: 이 프로젝트만의 플러그인 상태 (~/.agent/profiles/<projectId>/<tool>)
agentpod run --profile bot       # 명시: 여러 프로젝트가 "bot"이라는 이름으로 플러그인 상태를 공유/복사
# 크레덴셜: ~/.agent/profiles/bot/<tool>   (컨테이너 이름도 agent-<id>--p--bot)

--profile(또는 AGENT_PROFILE)은 run/shell/stop/rm/export에 적용됩니다. 같은 --profile 이름을 여러 프로젝트에서 쓰면 그 프로젝트들끼리는 플러그인 설치 상태를 의도적으로 공유/복사하는 효과가 있습니다. 이 자동 분리는 claude뿐 아니라 codex/opencode 크레덴셜에도 동일하게 적용됩니다(로그인 공유 예외는 claude의 .claude.json에만 해당).

mise — 툴체인 버전

이미지에 mise + python 3.12 + node가 내장됩니다. 프로젝트 루트에 mise.toml을 두면 부팅 시 그 버전을 추가 설치합니다.

# 프로젝트 루트/mise.toml
[tools]
python = "3.12"
node = "22"

5인증 — 두 가지 방식 (둘 다 지원)

① API 키

프로젝트 루트에 .env를 만들고 키를 넣습니다. 실행 시 자동 주입됩니다.

# .env  (gitignore 됨)
ANTHROPIC_API_KEY=sk-ant-...

.env.example을 복사해 시작하세요.

② 대화형 로그인 (구독)

컨테이너 안에서 한 번만 로그인하면 호스트에 저장되어 컨테이너가 죽어도, 다른 프로젝트로 옮겨도(§ 프로파일 참고 — 로그인은 프로젝트별로 분리되지 않고 항상 공유) 그대로 유지됩니다.

agentpod shell
# 컨테이너 안에서
claude login
exit

둘이 공존하면 API 키가 우선합니다.

6컨테이너별 MD 컨텍스트 만들기

컨테이너마다 다른 지침·문서를 보게 하려면, 그 컨테이너의 컨텍스트 폴더에 .md 파일을 넣기만 하면 됩니다. 코드 수정 불필요.

  1. 폴더 경로 확인 
    # 프로젝트 디렉토리에서
agentpod context
# → ~/.agent/contexts/<projectId> 경로 출력
  2. 지침 파일 작성 — 그 폴더에 CLAUDE.md를 만들고 에이전트 지침을 씁니다. 참고 문서도 *.md로 함께 둘 수 있습니다. 
    ~/.agent/contexts/<projectId>/
├── CLAUDE.md      # 자동으로 세션에 주입됨
└── design-notes.md
  3. 실행 — 다음 agentpod run부터 CLAUDE.md의 지침이 자동 반영됩니다(컨테이너 유저 메모리에 @import로 연결).
독립성: 폴더 하나 = 컨테이너 하나입니다. 다른 컨테이너의 MD를 상속하거나 참조하지 않습니다. 당신 저장소 파일은 전혀 건드리지 않습니다.

7여러 컨테이너 다루기

각 프로젝트는 경로로 결정된 고유 컨테이너를 갖습니다. 여러 프로젝트를 오가며 자연스럽게 여러 컨테이너를 운용할 수 있습니다.

cd ~/work/api      && agentpod run     # agent-api-xxxx
cd ~/work/frontend && agentpod run     # agent-frontend-yyyy

agentpod status                         # 전체 목록·상태·세션 수
cd ~/work/api      && agentpod stop     # 특정 프로젝트만 정리

같은 프로젝트로 다시 들어가면 동일 컨테이너가 재사용되어 세션·컨텍스트가 이어집니다.

8공유하기 — agentpod export

세션 중에 설치한 플러그인/스킬, 붙인 MCP 서버는 기본적으로 이 프로젝트 폴더만의 격리된 상태(§ 프로파일)에 남습니다 — git clone해도 다른 사람에게 자동으로 안 따라옵니다. agentpod export가 지금 상태를 프로젝트 파일로 스냅샷해줍니다:

# 프로젝트 폴더에서
agentpod export
# → agent.toml에 [[skills]] 추가, .mcp.json에 mcpServers 추가
# → 시크릿(MCP env/헤더 값)은 .mcp.json에 안 박히고 .env(gitignore)로 분리

이후 git diff agent.toml .mcp.json으로 확인하고 커밋·푸시하면, 팀원은 git clone + agentpod run만으로 같은 스킬이 자동 설치되고 MCP 서버가 자동 승인됩니다(.mcp.json의 ${VAR}가 참조하는 값은 각자 자기 .env에 채워야 함). 자세한 스키마·문제 해결은 스킬 가이드 §6 참고.

9문제 해결

증상원인 / 해결
Docker is not availableDocker 데몬이 꺼져 있음. Docker Desktop/데몬을 켜고 재시도. WSL2면 배포판 WSL Integration 확인.
WSL 안에서 docker: command not foundDocker Desktop → Settings → Resources → WSL Integration에서 해당 배포판 토글 ON.
컨테이너 안 claude가 로그인 요구API 키(.env)를 넣거나 agentpod shell → claude login.
빌드가 느림 (최초)정상. ubuntu·Node·claude 설치로 몇 분 소요. 이후 캐시로 빨라짐.
마운트 파일 권한 오류호스트 사용자 uid가 1000이어야 컨테이너 agent(uid 1000)와 정합. WSL2 기본 사용자는 대개 uid 1000.
컨테이너가 안 멈춤agentpod status로 활성 세션 확인 → 남은 세션이 있으면 유지됨. 강제 정리는 agentpod rm.

10안전 수칙

자율 실행 플래그는 컨테이너 안에서만. --dangerously-skip-permissions는 AgentPod가 컨테이너 내부에서만 사용합니다. 호스트에서 직접 이 플래그로 claude를 실행하지 마세요.
  • .env는 커밋하지 마세요(기본 .gitignore에 포함). 비밀 값을 이미지·저장소에 하드코딩하지 마세요.
  • 봇/전용 계정으로 최소 권한만 부여하고, 토큰·키는 주기적으로 회전하세요.
  • 런처(호스트)를 실행할 수 있는 사용자만 에이전트를 구동할 수 있도록 OS/도커 접근을 통제하세요.
